# v1.5 — Ledger & Write-offs Design

**Date**: 2026-05-13
**Status**: Design approved, awaiting written-spec review
**Target release**: v1.5 (post v1.4 stable cut)
**Tracking**: GitHub `#72` (umbrella); to split into 4 sub-issues at implementation start

---

## 1. Scope & Success Criteria

v1.5 closes the cost ↔ payment ↔ spend triple. v1.4 froze cost and per-player owed amounts at settle time; v1.5 makes the resulting gap visible to the organizer and makes "covering" a first-class action.

### In scope
- Dedicated admin-only `/admin/ledger` page with configurable date range and toggle between "By session" / "By player"
- Inline "Cover their $X" action on the existing PaymentsCard per-player action sheet
- Confirm sheet on remove-after-settle that offers "Cover & remove" as the default friendly path
- Reconciliation math: covering acts like an organizer-paid synthetic payment — collected grows, gap shrinks
- One new read endpoint: `GET /api/admin/ledger?from=&to=`

### Out of scope
- Player-facing visibility (admin-only)
- Bulk "mark all caught up" action
- Reset / zero-out baseline (date range filtering substitutes)
- Schema changes — `Player.writtenOff` already exists from v1.4 (additive)
- Multi-org support (see §7)

### Success criteria
1. Admin can answer "did Anna ever pay me back?" in two taps from Command Center
2. Admin can clear Bruce's debt without removing him from history
3. Admin who removes a player post-settle is prompted, not silently leaving orphan debt
4. Ledger math holds: `collected + covered = total paid`; `total cost − total paid = gap`

## 2. Architecture

### Endpoint: `GET /api/admin/ledger?from=ISO&to=ISO`

- **Auth**: `isAdminAuthed(req)` → 401 otherwise
- **Defaults**: `from` = 12 weeks ago, `to` = now if omitted
- **Validation**: ISO 8601, `to > from`, range ≤ 730 days, `from` not in future
- **Settled filter**: only sessions where `session.settled` is set. Un-settled sessions excluded from `spent` and `bySession` so admins understand "your gap reflects bills already sent"

**Response shape:**
```ts
{
  range: { from: string; to: string };
  summary: {
    spent: number;          // sum of settled session.totalCost in range
    paidAmount: number;     // sum of player.owedAmount where paid===true
    coveredAmount: number;  // sum where writtenOff===true
    collected: number;      // paidAmount + coveredAmount
    gap: number;            // spent − collected
    sessionCount: number;   // settled sessions in range
    coveredCount: number;   // players with writtenOff===true in range
  };
  bySession: Array<{
    sessionId: string;
    date: string;            // ISO
    attendanceCount: number;
    totalCost: number;
    paidCount: number;
    coveredCount: number;
    unpaidAmount: number;
    unpaidCount: number;
  }>;
  byPlayer: Array<{
    memberId: string | null; // null for legacy pre-migration records
    name: string;
    sessionCount: number;    // unpaid sessions in range
    owedAmount: number;      // still owed (paid=false && writtenOff=false)
  }>;
}
```

### Settled predicate
A player record counts as **settled** when `paid === true OR writtenOff === true`. The two flags are kept mutually exclusive by the setter (see Mutation below); the permissive `||` predicate is defensive against legacy records where both could be true.

### Mutation: existing `PATCH /api/players/[id]` extended
Adds `{ writtenOff: boolean }` to the accepted body. Same auth (admin-only for `writtenOff`; existing deleteToken path for `paid` is unchanged). Same response strip (`deleteToken`, `pinHash` removed).

**Setter rule**: setting `writtenOff: true` also sets `paid: false` and vice versa. Mutual exclusion lives in the route handler, not the client. If the client sends `{ paid: true, writtenOff: true }` in one body (race or buggy client), `writtenOff` wins — the route forces `paid: false`. Single-direction rule, no precedence puzzle.

### Soft-deleted players
`player.removed === true` does **not** exclude from ledger aggregation. The "Bruce moved away" scenario depends on it: removed from roster but unpaid `owedAmount` still surfaces until covered.

### Per-player merge
Aggregation joins records by `memberId` if present, else case-insensitive `name`. Same merged key sums into one `byPlayer` row. A record with a `memberId` and a same-name legacy record (no `memberId`) merge by name. Documented in the endpoint test.

### Drill-ins
- **Tap a session row** → existing PaymentsCard route for that session
- **Tap a player row** → existing `<PlayerProfileSheet>` from v1.4 (reuse, do not rebuild)

### Concurrency
Two admins covering the same player → last write wins on the server. Both ledger pages refetch on window `focus` and after every PATCH. Idempotent — a second cover of an already-covered player is a no-op write.

### Query cost
The aggregation runs a cross-partition scan of `players` (partitioned by `/sessionId`) joined with `sessions` (also `/sessionId`).

**v1.5 single-org scale (expected):**
- Worst case ~700 player records across ~52 sessions in a 12-week range
- Cross-partition scan completes in ~100-200ms on Cosmos serverless tier
- ~50-100 RU per query, well within free-tier daily budget at expected query rate

**Stage-2 multi-org future:**
- Cross-partition cost grows linearly with player count across all orgs
- Mitigation deferred: add `orgId` filter at migration time + consider denormalizing aggregate counters onto `sessions`. Out of scope for v1.5; flagged so the next scale review doesn't have to re-derive it.

No new indexes needed — `/sessionId` partition key already exists on both containers.

### Components (new files only)
```
app/api/admin/ledger/route.ts        ← new endpoint
app/admin/ledger/page.tsx            ← new Next route
components/admin/LedgerPage.tsx      ← page component, list logic inline
components/admin/CoverSheet.tsx      ← unified cover sheet (cover-only + cover-and-remove modes)
```

Inline-by-default: summary tiles, session list, player list, date range chips all live in `LedgerPage.tsx` until duplication forces extraction. The existing `<BottomSheet>` per-player action sheet in PaymentsCard gets a new `<ActionRow icon="paid" label="Cover their $X" />` row — no new wrapper, just a fourth item alongside Remove / Reset / Clear PIN.

## 3. UI Surfaces

### Ledger page (`/admin/ledger`)

**Date range UX** — 4 preset chips at the top: `All time` / `Last 12 weeks` (default) / `30 days` / `Custom…`. Custom opens a date-range picker. (Note: the existing `<DatePicker>` primitive is single-date; the range picker is incremental design work captured under Sub-issue B.)

**Default range** — Last 12 weeks. Weekly sessions × 12 = a manageable mental window, catches lingering debts, "All time" is one tap away. New friend groups with <12 weeks of data see the same range and an empty state.

**Summary header** — three tiles (Collected / Spent / Gap) with one sub-line: *"$8 covered by you · 12 sessions settled · 2 still owing"*. The covered callout keeps the headline `collected` number honest. (Multi-admin phrasing — "covered by us" or "covered (by Grant)" — is a known polish gap; punted to a follow-up unless someone surfaces it.)

**Toggle** — segment control (`.segment-control` class). "By session" is default (most admin trips start with "what about last week?").

**Friend-voice copy**
- Page subtitle: *"Who owes what. Cover anyone you don't want to chase."*
- Cover-only primary button: **"I got it"**
- Cover-and-remove primary button: **"Cover & remove"**
- Session row settled state: ✓ *all settled* (paid + covered both count)
- Sub-line phrasing: *"$8 covered by you"*

**Empty states**
- No sessions in range → *"Nothing settled in this window. Try widening the range."*
- All settled (gap = $0) → *"Everyone's caught up. Nice."*

**Drill-ins**
- Tap session row → existing PaymentsCard route (no new component)
- Tap player row → existing `<PlayerProfileSheet>` from v1.4 (no new component)

### PaymentsCard integration

The existing per-player `<BottomSheet>` (opened by the `⋯` button) gains a fourth `<ActionRow>`:

```
[icon=paid]  Cover their $8
[icon=person_remove]  Remove from session
[icon=key]            Generate recovery code
[icon=lock_clock]     Clear PIN
```

Tap → chains to `<CoverSheet mode="cover-only" />` for the friend-voice confirm step (a separate BottomSheet because the friend-voice copy "I got it" doesn't fit `window.confirm`).

### CoverSheet component

One polymorphic component with two modes:

| Mode | Trigger | Primary button | Secondary | Math |
|---|---|---|---|---|
| `cover-only` | PaymentsCard ActionRow | **I got it** | Cancel | `PATCH writtenOff:true` |
| `cover-and-remove` | RosterPage Remove (when `owedAmount > 0 && !writtenOff`) | **Cover & remove** | **Remove without covering** + Cancel | If "Cover & remove": `PATCH writtenOff:true` → `PATCH removed:true`. If "Remove without covering": `PATCH removed:true` only. |

~80 lines. Shared body, button row varies.

## 4. Error Handling & Edge Cases

### Network failures
- **`GET /api/admin/ledger` fails** → page renders explicit error state (*"Couldn't load the ledger — tap to retry. (Backend may be cold-starting.)"*). Per CLAUDE.md *"Lying empty state is forbidden"* — never `loaded` with empty arrays unless the API returned 200 with empty arrays.
- **`PATCH /api/players/[id]` (cover) fails** → optimistic update reverts, toast *"Couldn't cover — try again."* No silent swallowing.

### Concurrency
Last write wins. Client refetches on focus + after every own PATCH. Brief stale-state window (admin B sees uncovered after admin A covers, until B's next focus event) is acceptable; B's optimistic cover succeeds harmlessly as an idempotent no-op write.

### Validation
- `to ≤ from` → 400 (client UI prevents)
- Range > 730 days → 400 (matches Cosmos point-in-time restore window — anything older isn't restorable anyway)
- `from` in future → 400
- Non-admin caller → 401

### Edge cases

| Scenario | Behavior |
|---|---|
| Player has `paid: true` AND `writtenOff: true` (legacy / race) | Counted as paid once. Permissive `paid \|\| writtenOff` predicate prevents double-count. |
| Player has `owedAmount: 0` (settled-at-zero) | Excluded from `byPlayer`. Still counts toward `attendanceCount` in `bySession`. |
| Session not settled in range | Excluded from `spent` + `bySession`. Their players' un-frozen amounts aren't in the math. |
| `removed: true` player with unpaid `owedAmount` | **Included** in `byPlayer`. The Bruce-moved-away scenario depends on this. |
| Legacy player (no `memberId`) | Merged with same-name records by case-insensitive name. Documented in test. |
| Same name across orgs (post Stage-2) | Handled by `orgId` scope at Stage-2 migration time. Not v1.5's concern. |

### Rate limiting
- `GET /api/admin/ledger` — `checkRateLimit(`ledger:${ip}`, 30, 60_000)` (30/min). Generous — legitimate refresh-on-focus is normal admin behavior.
- `PATCH /api/players/[id]` — existing rate limit applies, no change.

### Known limitations (documented, not addressed)
- **No "officially caught up" state stored.** Date-range filtering isn't a substitute for "I tallied at Christmas, ignore everything before." A new device sees the full history regardless of offline reconciliation. Acceptable for v1.5; surface for future iteration.
- **No undo on cover.** To uncover, admin finds the player in PaymentsCard and toggles paid back to false (which clears `writtenOff` via mutual exclusion). Acceptable for v1.5; an undo toast can come later if friction surfaces.
- **Multi-admin "covered by you" copy.** Single-admin-centric. Polish gap; surface if it bites.

### Issue v1.5/D — Command Center entry + drill-in wiring

- Wire ledger session-row tap → PaymentsCard for that session
- Wire ledger player-row tap → existing `<PlayerProfileSheet>`
- Integration tests for navigation
- **~100 lines · depends on B · finishes the loop**

### Dependency graph
```
A (write-off PATCH + ActionRow)  ──┐
                                    ├─►  C (remove-after-settle prompt)
B (ledger endpoint + page)       ──┘
                                    ├─►  D (CC entry + drill-ins)
```

A and B can land in parallel. C and D follow.

### Recommended order
1. **A** — smallest, ships standalone value
2. **B** — biggest, headline feature, unblocks D
3. **D** — cheap finish, makes B reachable
4. **C** — completes the trio; depends on A

### Flag-gating
All four sub-issues land behind `NEXT_PUBLIC_FLAG_LEDGER`. Default off in stable, on in bpm-next. Promoted to stable when v1.5 cuts. Flag retired 2 weeks after stable cut per CLAUDE.md retention rule.

### Sizing
~1000 lines total production code (corrected from initial 850 estimate), ~25-30 new tests, four PRs over 2-4 weeks calendar time depending on review velocity. Schema migration cost: zero.

## 7. Multi-org Scope Statement

v1.5 targets the single-org friend-group deployment. Multi-org support is the Stage-2 migration playbook, separately tracked. v1.5 design assumes implicit single-org scope throughout (no `orgId` field, no per-org filtering); Stage-2 will coordinate adding `orgId` across all admin endpoints in one migration. Nothing in v1.5 blocks that future migration.
